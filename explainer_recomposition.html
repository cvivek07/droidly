CounterScreen.kt
@Composable fun CounterScreen() { var count by remember { mutableStateOf(0) } // ↑ state change triggers recomposition Text(text = "Count: $count") Button(onClick = { count++ }) { Text("Increment") } }
mutableStateOf
Compose observes this — any write triggers a re-run of this composable
remember { }
Survives recompositions — without it, state resets to 0 every time
Stable vs Unstable types
❌ Unstable
data class User(
val items: List<String>
)
// List is unstable!
// Always recomposes
✅ Stable
data class User(
val items: ImmutableList<String>
)
// Immutable = stable
// Skips if unchanged
❌ Lambda creates new ref
Button(onClick = { doIt() })
// new lambda each time
✅ Stable reference
val onClick = remember { { doIt() } }
Button(onClick = onClick)
OptimisedList.kt
// derivedStateOf — only recomposes when result changes val isValid by remember { derivedStateOf { text.length > 3 } } // key() — forces recomposition with new identity items.forEach { item -> key(item.id) { ItemRow(item) } }
derivedStateOf
Recomposes only when the derived value flips — not on every keystroke
key(item.id)
Tells Compose which item is which — prevents mismatched reuse in lists
Key Takeaways 🎓
🔄
State change is the only trigger — Compose only reruns functions that read changed state
⚡
Use stable types (data classes, ImmutableList) so Compose can skip unchanged composables
🎯
derivedStateOf for computed values, key() for list items — both reduce unnecessary recompositions
